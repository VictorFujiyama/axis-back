# Changelog

## Phase D Builtin — Auto-provision atlas_user_links via Phase 0 SSO — 2026-05-13

- `atlas_user_links` is now auto-populated for every Atlas user that loads `/messaging` — no SQL INSERT, no admin UI, no support ticket. The Atlas-side `signAxisIframeToken` (Phase 0 SSO mint path) now embeds `atlas_app_user_id` (Clerk user id) + `atlas_org_id` (Atlas org UUID resolved via `getOrCreateUserOrg`) when both are present, and the axis-back `verifyAtlasIframeTokenWithSecret` round-trips them through the JWT verify step. The `/api/auth/exchange-iframe-token` handler then `INSERT ... ON CONFLICT DO NOTHING` on `(accountId, atlasOrgId, atlasAppUserId)` so the link row exists before the user's next MCP tool call. The very first iframe load is the implicit provisioning event.
- "Both or neither" invariant on the JWT (`src/plugins/atlas-iframe-auth.ts` verifier): if only one of `atlas_app_user_id` / `atlas_org_id` is present (or either is empty), both are dropped from the returned shape. Phase 0 SSO tokens without either field continue to verify cleanly (backward compat — see L-518). Validation is `typeof === 'string' && length > 0` only; `atlas_user_links.atlasOrgId` is `text` not `uuid`, so no UUID shape check.
- UPSERT is non-blocking: a DB error during `exchange-iframe-token` is logged via `app.log.warn` and the response still returns 200. The link is idempotent on the unique constraint, so the next iframe load retries naturally.
- `requireAtlasUserLink` error string in `src/modules/atlas-mcp/tools.ts` rewritten from the internal-sounding "no atlas_user_link for this Atlas user in this account" to the actionable "Atlas user not linked — open /messaging in Atlas web first to activate the link, then retry." The string bubbles through `MessagingToolError → MCP transport JSON → Atlas chat`, so end users see the fix path verbatim. The pre-existing `tools.spec.ts` case was strengthened to pin the exact wording.
- `FastifyRequest` interface gained `atlasIframePayload?: AtlasIframePayload`; the `requireAtlasIframeAuth` decorator now sets `req.atlasIframePayload = payload` alongside the existing `atlasIframeUser` decoration so downstream handlers can read both ids without re-decoding. Existing callers that only read `atlasIframeUser` are unaffected.

## Phase D Activation — Bearer auth + envelope feature flag — 2026-05-12

- Bearer token auth via `MCP_AXIS_API_KEY` env (Phase 11 Atlas-side `mcp_servers.headers` compat — `{"Authorization":{"ref":"env://MCP_AXIS_API_KEY"}}`). `verifyMcpRequest` validates `Authorization: Bearer <hex>` with `crypto.timingSafeEqual` after a length-equal early-return (see L-507) — constant-time compare prevents byte-by-byte timing attacks on the static token.
- `MCP_AUTH_MODE` triple-mode switch (`hmac` | `bearer` | `both`, default `both`). `both` accepts either header during migration; `bearer` is the post-migration steady state; `hmac` is the rollback escape hatch (kill switch — see L-505: not graceful, because Atlas-side `mcp_servers.headers` won't flip back to `X-Atlas-Signature` without a registry edit + restart). Boot precheck enforces required envs per mode (Bearer+API-key, HMAC+secret); mode `both` with HMAC secret unset logs a warn and boots Bearer-only (T-003 decision).
- `USE_PHASE_12_ENVELOPE` feature flag — listeners default to the Phase B legacy shape (`{type:'message_sent'|'handoff_to_human'|'conversation_resolved'}`) until the Phase 12 connector receiver lands Atlas-side (see L-420). Flip the flag without a redeploy when `/api/connectors/messaging/events` is live. Worker `serializeJob` dual-shape narrowing handles both shapes during the drain window.
- `ATLAS_EVENTS_ENDPOINT` default flipped from `/api/connectors/messaging/events` → `/api/messaging/events` (the Phase 12 receiver doesn't exist Atlas-side yet; the Phase B endpoint is the one running in prod). Render env overrides without code change once Phase 12 ships.
- HMAC code path retained behind `tryHmacAuth` as the kill-switch escape hatch (see L-505 — full graceful rollback also requires flipping Atlas-side `mcp_servers.headers` back to `X-Atlas-Signature`, which is out of scope for this phase). HMAC tests (5 sacred cases) preserved verbatim; auth suite grew 5 → 11 cases covering Bearer/mode combinations.
- Pre-existing test failures (`worker.spec.ts` x2, `atlas-routes.spec.ts` x2) carry over from prior phases and are unrelated to Phase D Activation — they predate this branch (the `ATLAS_EVENTS_ENDPOINT` default flip surfaced two of them; the other two are T-018 DB-mock drift).

## Phase D.3 — Atlas MCP write tools + atlas-bot user — 2026-05-12

- New `users.is_atlas_bot boolean NOT NULL DEFAULT false` column (migration `0028_secret_dragon_lord.sql`) gates the bot identity at the schema level. `src/modules/atlas-mcp/atlas-bot.ts` lazily provisions one `atlas-bot+<accountId>@atlas.internal` user per Axis account (name `Atlas Assistant`, `is_atlas_bot=true`, random 32-byte `passwordHash` — never used for login) plus an `account_users` row with `role='agent'` + `auto_offline=false`. Idempotent under concurrency via `ON CONFLICT DO NOTHING` on both the unique `users.email` and `(account_id, user_id)` constraints; second caller re-selects by email.
- `RealtimeEvent` union (`src/realtime/event-bus.ts`) gained an optional `meta?: {atlasAppUserId?, atlasOrgId?}` field on `message.created`, `conversation.assigned`, `conversation.resolved`. MCP write tools emit events with `meta` populated; the atlas-events `buildEnvelope` listener reads `event.meta?.atlasAppUserId` and stamps `actors[].app_user_id` on the Phase 12 §12.1 envelope. Phase A/B/C emitters keep emitting without `meta` (optional field, no breakage) — those events correctly produce envelopes with no `app_user_id` because they originate from humans/non-Atlas bots.
- Three MCP write tools registered via `buildMcpServer(db, app, ctx)` in `src/modules/atlas-mcp/server.ts`: `messaging.send_message`, `messaging.assign`, `messaging.resolve`. The Fastify plugin (`src/plugins/mcp-server.ts`) reads `X-Atlas-App-User-Id` + `X-Atlas-Org-Id` headers per request and passes them as `ctx` into the server factory; the factory captures `ctx` in closure (the SDK callback signature can't accept extra args) and a per-request `McpServer` instance keeps identities isolated. Missing identity headers surface as a `forbidden` tool error from write tools while read tools continue to work.
- Authorization gate: every write tool calls `requireAtlasUserLink(db, accountId, ctx)` and 403s if no row exists in `atlas_user_links` matching `(accountId, atlasOrgId, atlasAppUserId)`. `send_message` and `resolve` also call `getOrCreateAtlasBotUser` so the inserted row / `resolvedBy` is stamped with the bot user. `assign` validates that the target `userId` is a member of the conversation's inbox via `inbox_members` (mirroring `conversations/routes.ts`). `resolve` refuses if the conversation is already resolved and fires the CSAT prompt enqueue when `inbox.config.csat.enabled` (dynamic import, best-effort — mirrors the existing route precedent).
- `assignInputSchema` is split into a raw `ZodObject` + a refined `ZodEffects`: the SDK registers the raw shape (`assignInputObjectSchema.shape`) and the server wrapper applies `assignInputSchema.parse(args)` to enforce the "at least one of userId or teamId" rule. `ZodError` maps to `{isError:true, content:[{type:'text', text: JSON.stringify({error:'bad_request', message})}]}` for a clean LLM-visible failure.
- Tests: `atlas-bot.spec.ts` (4 cases — create, fetch existing, ON CONFLICT race recovery, account_users race), `tools.spec.ts` adds 5 write-tool cases (send happy / missing link 403, assign happy / non-member 400, resolve), `server.spec.ts` exercises tool registration end-to-end over `InMemoryTransport.createLinkedPair()` per L-419.

## Phase D.2 — Atlas MCP server (read tools) — 2026-05-12

- Added `@modelcontextprotocol/sdk@^1.29.0` dependency, aligned with the Atlas-side runtime (Berg `phase-11-12` resolves the same minor). Plus the `atlas_user_links` schema in `packages/db/src/schema/atlas-user-links.ts` (migration `0027_wealthy_legion.sql`) maps Atlas `(orgId, appUserId)` to an Axis user per account; FKs cascade from `accounts` / `users`, unique on `(accountId, atlasOrgId, atlasAppUserId)`.
- Config gating in `src/config.ts`: `MCP_SERVER_ENABLED` (enum-then-transform, default `false`) and `ATLAS_MCP_HMAC_SECRET` (min 16, optional). Boot precheck throws if MCP is enabled without the secret — fail-fast instead of silent 401s. The MCP HMAC secret is intentionally separate from `ATLAS_EVENTS_HMAC_SECRET` (inbound MCP vs outbound events), but Felipe can point both at the same value on Render without code changes.
- Fastify plugin `src/plugins/mcp-server.ts`: when enabled, registers `POST /mcp` with a plugin-scoped `application/json` content-type-parser that captures `request.rawBody` as a Buffer (no leak to other routes — encapsulation, no `fastify-plugin` wrap). PreHandler verifies `X-Atlas-Signature` via `verifyMcpRequest` (reusing the Phase B `verifyOutboundSignature` Stripe-style primitive) → 401 on missing/invalid. Per-request `McpServer` + stateless `StreamableHTTPServerTransport` (`sessionIdGenerator: undefined`) with `reply.hijack()` + idempotent cleanup on `reply.raw.on('close')`.
- Tool registrations in `src/modules/atlas-mcp/server.ts` via `buildMcpServer(db)`: `messaging.get_thread` (UUID input, 50-message window chronological), `messaging.list_threads` (inboxId / status / assignee / since / limit, soft-deleted excluded, desc `updatedAt`), `messaging.search` (tsvector `messages.search_vector` + `ts_rank`). Handler errors surface as `{isError:true, content:[{type:'text', text: JSON.stringify({error})}]}` so the LLM sees the failure cleanly. Dot-notation logical names; Atlas-side `registry-bridge.ts` prefixes to `mcp_<orgId>_messaging_<tool>` per Phase 11 precedent.
- Tests cover the data path (`src/modules/atlas-mcp/__tests__/tools.spec.ts` — 6 cases on mocked db chains), the MCP wire path (`server.spec.ts` — `InMemoryTransport.createLinkedPair()` listTools + happy + isError), and the HTTP/HMAC gate (`src/plugins/__tests__/mcp-server.spec.ts` — disabled→404, missing/bogus sig→401, valid sig+initialize→200).

## Phase D.1 — Atlas connector envelope (Phase 12 §12.1) — 2026-05-12

- AtlasEventJob expanded to the `kind`/`action` discriminator envelope (`conversation_turn` | `conversation_summary` | `contact`). Listeners in `src/modules/atlas-events/enqueue.ts` now build envelopes via the pure helpers in `src/modules/atlas-events/build-envelope.ts` (`buildConversationTurnEnvelope` / `buildHandoffEnvelope` / `buildResolvedEnvelope`), populating `actors[]` / `participants[]` / `viewableBy` / `accountId` and capping `summary` to 200 chars. Phase B legacy variants stay in the union to drain in-flight queue jobs during deploy.
- Worker (`src/modules/atlas-events/worker.ts`) gained a dual-shape serializer: new envelopes are POSTed as snake_case (`source_ref`, `occurred_at`, `account_id`, `viewable_by`, `actors[].app_user_id`) while legacy Phase B jobs keep their existing wire shape. Endpoint is now configurable via `ATLAS_EVENTS_ENDPOINT` (default `/api/connectors/messaging/events` — Phase 12 receiver; rollback to `/api/messaging/events` by env override without redeploy).
- Idempotency uses `sourceRef` as BullMQ `jobId` (`<conversationId>:<event>:<messageId|timestamp>`), so retries and duplicate emits dedupe deterministically at enqueue time.
